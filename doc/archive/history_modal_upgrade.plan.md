# History modal upgrade — slim header, delineated items, model-named sessions

## Background

The Conversation History modal (screenshot-reviewed 2026-06-04) has three
problems:

1. **The header is far too tall** — a single line of title text sits in a bar
   with `16px 20px` padding; the user wants it ~half height, slim.
2. **Rows don't read as discrete items** — at rest there is no border, no
   background, no divider; the only per-row distinction is the `:hover`
   background, so the list looks like one undifferentiated gray column with too
   much whitespace.
3. **Session names are weak** — the label is just the verbatim first user
   message (lightly cleaned). Many start identically ("Hello, what model am I
   talking to?", "Hello, is anyone there?"), so rows are hard to tell apart.

Separately, the index is currently **capped at 50** conversations
([conversation.ts](../src/conversation.ts) ~342), but the modal should be
designed to stay scannable at **~100**.

Decisions taken with the user before drafting:
- **Slim the shared header** (changes Settings/Tools too — accepted for
  consistency), not a History-only override.
- **Include title generation in this plan** (not a separate plan).
- **Generate titles via a model query** (not a pure heuristic).

## Approach

Two coordinated tracks, shippable together.

**Track A — Visual (CSS + one cap bump).** Confined to
[Modal.less](../src/webview/components/Modal/Modal.less) and
[ConversationHistory.less](../src/webview/components/ConversationHistory/ConversationHistory.less),
plus raising the index cap in
[conversation.ts](../src/conversation.ts). Halve the shared header, add
divider lines between rows, ensure a clear hover background, and tighten the
row rhythm so ~100 rows scroll cleanly inside the modal body (which already has
`max-height: 90vh` + `overflow-y: auto`).

**Track B — Model-generated titles.** Persist a new optional `title` on the
conversation and its index entry; render `title || firstUserMessage`. Generate
the title with the model **in-session via `sendSilentQuery`**, on a **3-then-6
turn schedule**.

> **Verified in code — `sendSilentQuery` does NOT pollute the transcript.**
> Tracing [subprocess.ts](../src/subprocess.ts): the question is written
> straight to the CLI's stdin (~138), **bypassing** `sendAndSaveMessage`, so it
> never enters `currentConversation` and never posts to the webview. The answer
> is caught by `silentQueryCallback` (~1071-1074) which fires and `continue`s,
> **also bypassing** `sendAndSaveMessage`. **Result:** the visible transcript is
> untouched and `messageCount` (= `currentConversation.length`) is **not**
> bumped. The only caveat: the Q&A does live in the CLI's own `--resume`
> context, costing a small amount of context tokens going forward — acceptable.
>
> This makes `sendSilentQuery` the **right** tool (an earlier draft wrongly
> avoided it): the model is a live participant with full conversation context,
> so "knowing what we've covered, what would you call this?" yields a genuinely
> descriptive title. An out-of-band process would have no context unless we
> re-fed the whole transcript — more expensive and worse.

> **⚠️ Re-verify against the session-lifecycle Phase 2 work (control protocol +
> single-process reuse).** `sendSilentQuery` was speced before that landed; two
> things changed and MUST be handled (see todo `silentquery-onturnend-guard`):
>
> 1. **A silent query's `result` now drives `onTurnEnd()`.** Under the reuse
>    model, a silent title query produces a normal `result` event, and
>    `onTurnEnd()` (subprocess.ts ~896) does more than it used to: it disarms the
>    stall watchdog, **drains the queued-turn list** (`queuedTurns`), and
>    **applies a deferred `set_model` / settings restart** (`pendingModelSwitch`,
>    `pendingSettingsRestart`). So firing a title query at a turn boundary could
>    prematurely flush the user's queued prompt or trigger a deferred model
>    switch. **Fix:** mark a silent-query completion (e.g. an
>    `awaitingSilentResult` flag set on send, cleared in the callback) and have
>    `onTurnEnd` skip its drain/deferral path for that result.
> 2. **Single in-flight slot.** `silentQueryCallback` is one slot, not a map. With
>    forward-only titling (decided) there is never more than one outstanding
>    silent query, so the single slot is fine and queries are serialized. (Legacy
>    backfill — which would need 2–3 concurrent — is explicitly deferred; it would
>    require converting the callback to a keyed map.)
>
> **On the screenshot's high message counts (1710/4782):** these are NOT a bug
> and NOT caused by silent queries. `messageCount = currentConversation.length`
> counts every *saved message event* (each tool_use, tool_result, assistant
> chunk, etc.), not user turns — so large numbers for long sessions are expected.
> No action here; do not treat it as a defect.

**Why 3-then-6.** Titling off the *first* message is weak ("Hello, what model
am I talking to?" says nothing about where the chat went). Waiting lets the
topic reveal itself. But many chats are short, so:
- **At 3 user turns** → generate a **provisional** title (enough signal for most
  chats; covers the ones that end soon after).
- **At 6 user turns** → regenerate the **final** title and **lock it forever**
  (no further regeneration, even for marathon sessions — keeps it cheap and the
  list stable).
- Ending **before 3 turns** → no model call; keep the `cleanSessionTitle`
  heuristic.

**"Turn" means a `userInput` message**, counted explicitly — NOT
`currentConversation.length` (which counts every stream event and would fire the
trigger almost immediately).

### B1 — Persist a title

### B2 — In-session generation via sendSilentQuery (3-then-6)

**Order of operations matters here.** The title trigger must run at the idle
turn boundary, but it must NOT be tangled with `onTurnEnd`'s queue-drain /
deferred-switch logic (see the ⚠️ re-verify note). Sequence inside `onTurnEnd`,
*after* the silent-query guard has decided this is a real user turn (not a
silent-query completion):

```
// onTurnEnd, only for a REAL user-turn result (not a silent-query result):
maybeGenerateTitle();   // checks the 3-then-6 schedule
// ...then the existing drain/deferral logic...
```

Track user turns explicitly — `userTurns = currentConversation.filter(m =>
m.messageType === 'userInput').length` (NOT `currentConversation.length`):

```
if (titleLocked) return;
if (userTurns >= 6) { generateTitle(provisional=false); titleLocked = true; }
else if (userTurns >= 3 && !hasProvisionalTitle) generateTitle(provisional=true);
```

(Use `>= 6` not `=== 6` so a session resumed at turn 8 with no final title still
gets one. Likewise the provisional is `>= 3 && !hasProvisionalTitle`, NOT `=== 3`:
under the queue feature several turns can drain across boundaries before the title
query fires, so `userTurns` can jump 3→5 and an exact `=== 3` would silently skip
the provisional entirely. Check the final `>= 6` branch first so a session that
crosses both thresholds at once goes straight to the locked final title.
`titleLocked` is the guard against re-firing the final; `hasProvisionalTitle`
(true once a provisional has been stored) guards against re-firing the
provisional — see below.)

`generateTitle` calls `sendSilentQuery(TITLE_PROMPT, (answer) => { ... })`:
- Sanitize the answer (trim, strip wrapping quotes, drop trailing punctuation,
  clamp to ~80 chars, collapse to one line).
- If the answer is empty/junk, keep the existing title (provisional or
  heuristic) — never overwrite a good title with garbage.
- Store on the conversation (`conversationData.title`), set `titleLocked` when
  final, re-save, update the **index entry** (the list renders from the index),
  and `sendConversationList()` so the row updates live.

**`titleLocked` is a STORED boolean** (decided with user), persisted on both
`ConversationData` and the index entry. On reload it's read back directly — no
recomputing turn counts, no risk of re-firing the 6-turn generation on a resumed
locked session. A provisional-titled session resumed at turn 4 correctly still
fires the final generation when it reaches turn 6, because `titleLocked` is
still false.

**Proposed prompt** (`TITLE_PROMPT`):

> Quick side question, unrelated to the task at hand: I'm assembling a list of
> past conversations and need a short working title for this one. Knowing what
> we've covered so far, what would you call it? Reply with only the title —
> about 3 to 6 words, descriptive enough to tell it apart from other chats, with
> no surrounding quotes and no trailing punctuation.

The "reply with only the title" + explicit format constraints are load-bearing:
without them the model answers conversationally ("Sure, I'd call this…") and the
result needs heavy stripping.

**Timing safety:** `sendSilentQuery` no-ops to `callback('')` if there's no live
stdin, and queues via `pendingSilentQuery` if a turn is in flight — so firing it
at the turn boundary is safe and won't collide with the user's next message.

### B3 — Fallback (forward-only; no backfill in v1)

- **Fallback:** whenever `title` is absent or generation fails, render
  `firstUserMessage` (already cleaned via `cleanSessionTitle`, ~294). Nothing
  regresses — legacy and short (<3-turn) conversations simply keep the heuristic
  label.
- **Legacy backfill — DEFERRED (not in v1).** Titling untitled legacy
  conversations on modal open would require generating several titles at once.
  The current machinery has a **single `silentQueryCallback` slot** — only one
  silent query can be outstanding — so concurrent backfill isn't safe without
  first converting the callback to a keyed map (`Map<id, callback>`). That
  conversion is out of scope here. v1 is **forward-only**: new conversations get
  titles as they cross the 3- and 6-turn marks; everything already on disk keeps
  the heuristic. (No out-of-band processes are involved at all — titling is
  in-session via `sendSilentQuery`.)

## Edge cases

- **Transcript pollution:** verified that `sendSilentQuery` bypasses
  `sendAndSaveMessage` for both question and answer, so the title Q&A does NOT
  appear in the transcript and does NOT bump `messageCount`. Still, verify
  message count is unchanged before/after generation as a regression guard.
- **Title generation failure / offline / Bedrock hiccup:** `sendSilentQuery`
  calls back with `''`; on empty/junk, keep the existing title (provisional or
  heuristic). Never block saving or list rendering.
- **Junk/conversational answer** ("Sure, I'd call this…"): the prompt constrains
  to title-only; the sanitizer strips quotes/punctuation; if still unusable,
  fall back. Never overwrite a good title with garbage.
- **Provisional vs final race:** if the user is mid-turn when the 3- or 6-turn
  boundary hits, `sendSilentQuery` queues via `pendingSilentQuery` and fires at
  the next idle boundary — confirm the lock logic tolerates the slight delay.
- **Header slimming affects other modals:** Settings and Tools share the header
  — visually confirm they still look right (intended trade-off).
- **Index cap raise (50→100):** existing pruning logic is unchanged except the
  number; entries beyond 100 still get pruned (their JSON files remain on disk).
- **Concurrent silent queries (not in v1):** the title flow assumes at most ONE
  outstanding silent query (single `silentQueryCallback` slot). Forward-only
  titling guarantees that. If legacy backfill is ever added, the callback must
  first become a keyed map — otherwise a second query overwrites the first's
  callback and a title is silently dropped.
- **Cost:** each title is a tiny extra model call. Acceptable per user decision;
  keep prompts short and one-shot.

  marathon-session retitle tier was considered and deferred.
- **Not redesigning the modal into cards or adding accent borders** — the user
  chose subtle dividers + hover color only.
- **Not a History-only header override** — the user chose to slim the shared
  header for all modals.
- **Not changing the message-count display** — and NOT treating the high counts
  (1710/4782) as a bug: `messageCount = currentConversation.length` counts every

- **Not backfilling legacy untitled conversations** — forward-only for v1 (the
  single-slot silent query can't safely run concurrent backfill queries).
- **Not building concurrent silent queries** — out of scope; would require a

- **Not touching the AskUserQuestion-blocks-the-prompt-input bug** — that is a

## Resolved decisions

- **Schedule:** 3 user turns → provisional, 6 user turns → final + lock forever.
  (Decided with user.)
- **Generation method:** in-session `sendSilentQuery` (verified non-polluting),
  not out-of-band. (Decided with user.)
- **Backfill scope:** **forward-only for v1.** Title new conversations going
  forward; legacy/untitled rows render the heuristic. No legacy backfill — that
  would need 2–3 concurrent silent queries, which the single-slot
  `silentQueryCallback` doesn't support; deferred until/unless we convert it to a
  keyed-callback map. (Decided with user.)
- **`titleLocked` persistence:** **stored boolean** on the conversation + index
  entry. Survives reload with no recompute; unambiguous. (Decided with user.)
- **`onTurnEnd` collision:** RESOLVED by the `silentquery-onturnend-guard` todo
  plus the explicit ordering contract below.

## The `onTurnEnd` contract (RESOLVED — load-bearing)

`onTurnEnd` is already a contention point in shipped code (subprocess.ts ~896):
today it flips `isProcessing` off, disarms the watchdog, flushes a pending silent
query, applies a deferred `set_model`, applies a deferred settings restart, and
drains one queued turn. This plan adds title generation, and
`prompt_queue_ux_spec.md` adds UI over the *same* `queuedTurns` drain. So the
ordering must be a stated contract, not incidental. Decided order, for a **real
user-turn `result`**:

1. **Guard first.** If this `result` is a **silent-query completion**
   (`awaitingSilentResult`), do NONE of the steps below — clear the flag **here, on
   the result event** (NOT when `silentQueryCallback` fired — that ran earlier on
   the text block ~1072) and return. Silent queries (including title generation)
   must not drain the queue or apply deferred switches. **This guard is the actual
   correctness lever** — it makes a silent-query result safe to interleave with a
   real turn regardless of issue ordering.
2. `isProcessing = false`; disarm watchdog; post `setProcessing:false`.
3. **Title trigger** (this plan): count `userInput` turns; run the 3-then-6
   schedule via `sendSilentQuery`. This *issues* a silent query but doesn't block;
   its own `result` returns through step 1 and is ignored by the drain.
4. **Deferred `set_model`** (Phase 2): apply `pendingModelSwitch` if set.
5. **Deferred settings restart** (Phase 2): apply `pendingSettingsRestart` if set.
6. **Drain one queued turn** (Phase 1 / queue UI): shift + run the head of
   `queuedTurns`.

Rationale: the guard (1) is the correctness fix — and the only one that fully
closes the seam. `sendSilentQuery` does NOT set `isProcessing` (subprocess.ts
~131-138), so a title query issued at step 3 leaves `isProcessing` false; step 6
then drains a queued turn and writes that too — so a silent query and a real turn
can be in flight together. The guard makes that safe (the title `result` returns
through step 1 and is ignored). Deferred switch (4/5) before drain (6) so the
queued turn runs under the intended model/provider.

> **Alternative considered — drain-first + defer the title via
> `pendingSilentQuery`.** `onTurnEnd` already calls `flushPendingSilentQuery()`
> (subprocess.ts ~901) and `sendSilentQuery` already defers when `isProcessing` is
> true (~127). So one could drain first: the drained turn sets `isProcessing=true`,
> the title query defers, and a later `onTurnEnd` flushes it. This SHRINKS the
> overlap window but does NOT eliminate it — under a multi-item queue, that later
> `onTurnEnd` flushes the title (still without setting `isProcessing`) and then
> drains the NEXT queued turn at step 6, recreating the overlap. The guard is
> required either way; deferral is at most a minor latency nicety, not a substitute.
> Keep the issue-at-step-3 order above for simplicity; correctness rests on the
> step-1 guard, not on ordering.

`prompt_queue_ux_spec.md` references this same
contract; both features must be implemented and tested against this exact order.

**Sequencing:** the `queuedTurns` + `onTurnEnd` drain ALREADY shipped (Phase 1/2);
the prompt-queue spec is mostly UI over existing machinery. So this title plan
must coexist with the drain **regardless of whether the queue UI ships**, and the
step-1 guard is required either way.
